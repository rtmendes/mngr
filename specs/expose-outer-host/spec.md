# Expose outer host (work in progress)

This spec is being authored via `/architect`. The Q&A phase is in progress;
the final spec will be generated by `/architect-generate`.

## Goal

Add a minimal, optional way to access the "outer" host of a `Host`
(container/sandbox), so that callers can run commands on the underlying
machine that hosts the container. `mngr exec` should be able to target
this outer host.

## Provider landscape

| Provider | Inner | Outer | Accessibility today |
|---|---|---|---|
| `local` | local | (same as inner / N/A) | trivial |
| `docker` (local socket) | container | local machine | shell-out locally |
| `docker` (DOCKER_HOST=ssh://...) | container | remote VM | SSH (`config.host` URL) |
| `docker` (DOCKER_HOST=tcp://...) | container | not generally accessible | none |
| `mngr_vps_docker` / `mngr_vultr` | container | VPS (root@vps_ip:22) | per-host SSH key on disk |
| `mngr_imbue_cloud` | container | leased VPS (root@vps_ip:22) | per-host SSH key on disk |
| `mngr_modal` | sandbox | N/A | none (by design) |
| `ssh` provider | the SSH host itself | (no outer) | none |

## Open design questions

1. Where the accessor lives and what it returns
   (`OnlineHostInterface.get_outer_host()` vs provider method vs new
   narrower interface).
2. Behavior for the `local` provider (`None` vs self).
3. Coverage for the `docker` provider's `ssh://` daemon URL in the first
   pass.
4. CLI surface — `mngr exec --outer <agent>` vs new verb vs address syntax.
5. Whether to migrate the existing one-off SSH paths
   (`mngr_imbue_cloud/vps_admin.py`, `mngr_vps_docker/docker_over_ssh.py`)
   in this PR or defer.
