# Common events for the "thinking" role 

The thinking agent is the primary role agent -- the "brain" of the mind. It receives events, decides how to respond, and delegates work to other role agents.

Below are the common event sources and types that the thinking agent receives, along with a brief description.
The agent itself will have access to more information about these events given its skills.

## Built-in event sources

The thinking agent receives events from the following built-in sources. Additional sources may exist depending on plugins and configuration -- use the `list-event-sources-and-types` skill for the current list.

All events share a common envelope: `{"timestamp", "type", "event_id", "source", ...additional fields}`.

### messages

User and assistant messages from conversation threads. Synced from the `llm` database by the conversation watcher.

Fields: `conversation_id`, `role` ("user" or "assistant"), `content`

User messages always arrive paired with the assistant response that was auto-generated by the talking agent. The thinking agent reviews the response and decides whether follow-up action is needed.

### mngr/agent_states

State transitions for agents (including sub-agents created via `delegate-task`). Written by the observer service.

Fields: `agent_id`, `state`, `data`

Possible states: `stopped`, `running`, `waiting`, `done`, `replaced`. See the `handle-mngr-agent_states` skill for how to respond to each.

### scheduled

Time-based trigger events. Fire at configured times (e.g., start of day, end of day, periodic checks).

Fields: `data` (payload describing the trigger)

### stop

Signals that the thinking agent is about to stop. This is the last chance to check for unprocessed work.

Fields: `data`

### conversations

Conversation lifecycle events (created, model changed). Primarily bookkeeping -- the `messages` source is more actionable.

Fields: `conversation_id`, `model`

### monitor (future)

Metacognitive reminders from a monitor agent. Not yet implemented.

### (logs)

Errors and warnings in the logs will show up as events from different sources that start with "logs/"

Fields: `level`, `message`
