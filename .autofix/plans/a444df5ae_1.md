# Fix: branch_to_check ordering in worktree error message

## What the issue is and where

In `libs/mngr/imbue/mngr/hosts/host.py`, line 1619, the error handling for "already checked out" git worktree errors uses:

```python
branch_to_check = base_branch or new_branch_name
```

This picks `base_branch` when both are set.

## Why it is a problem

When `git worktree add <path> -b <new_branch> <base_branch>` fails with "already checked out", git is referring to `new_branch` (the branch being checked out in the new worktree), not `base_branch` (which is just the starting point for the new branch). The current ordering would display the wrong branch name in the error message when both `new_branch_name` and `base_branch` are set.

For the common case (only `base_branch` set, no `new_branch_name`), this works correctly because `new_branch_name` is `None`. But for the case where both are present, the error message would reference the wrong branch.

## Planned fix

Change line 1619 from:
```python
branch_to_check = base_branch or new_branch_name
```
to:
```python
branch_to_check = new_branch_name or base_branch
```

This ensures the branch being checked out (which is what git's error refers to) is used in the error message.

## Risks and edge cases

- Low risk: this only changes behavior when both `new_branch_name` and `base_branch` are set, which is a less common path for this particular error.
- The existing test `test_worktree_already_checked_out_gives_helpful_error` only tests the case where `base_branch` is set and `new_branch_name` is None, so it won't be affected by this change.
