# Issue 1: Silent failure of mkdir/ln in batched symlink creation

## What the issue is and where

In `libs/mng/imbue/mng/hosts/host.py`, lines 1604-1632, the batched symlink creation script joins multiple shell commands with `;`. If `mkdir -p` or `ln -snf` fails for any symlink pair, the failure is silently ignored because:

1. The commands are joined with `;` (not `&&`), so execution continues to the next path
2. The overall exit code is determined by the last command in the script, which is `if [ -n "$errors" ]; then ... fi`
3. When `$errors` is empty (no CONFLICT entries), the `if` evaluates to false, and the shell exits with code 0
4. The Python code only checks `result.success`, so it does not detect mkdir/ln failures

## Why it is a problem

The original per-path code raised `MngError` when `mkdir -p` or `ln -snf` failed. The new batched code silently ignores these failures. A user could end up with missing symlinks and no diagnostic information about why.

## Planned fix

Modify the batched shell script to track mkdir/ln failures in addition to CONFLICT errors. Specifically:

1. After the `mkdir -p ... && ln -snf ...` command, add a check for the exit code and append an error message to the `$errors` variable if it failed.
2. This way, both CONFLICT errors and mkdir/ln failures are captured.
3. On the Python side, parse both CONFLICT and SYMLINK_FAILED prefixed lines from stderr.
4. Raise `UserInputError` for conflicts and `MngError` for other failures, maintaining the same error hierarchy as the original code.

## Risks and edge cases

- The fix adds a few more characters to each path's shell snippet, but this should be well within command length limits.
- Need to make sure the error message formatting doesn't introduce new shell interpretation issues (use printf for safety, consistent with the existing CONFLICT handling).
