# Issue 2: Mixed CONFLICT and FAILED errors only report conflicts

## What the issue is and where

File: `libs/mng/imbue/mng/hosts/host.py`, lines 1623-1630

When the batched symlink creation shell script encounters both CONFLICT errors (target exists
as non-symlink) and FAILED errors (mkdir/ln failed), the Python error handling only reports
the conflicts as a UserInputError and silently drops the FAILED messages.

## Why it is a problem

If a user has multiple extra paths configured and one has a conflict while another has a system
error (e.g., permission denied on mkdir), they would only see the conflict error. After fixing
the conflict and re-running, they would then see the system error -- a frustrating experience.

## Planned fix

Include FAILED paths in the error message when both types of errors occur. Specifically:
- Extract both CONFLICT and FAILED lines from stderr
- If only conflicts: raise UserInputError (current behavior)
- If only failures: raise MngError (current behavior)
- If both: raise UserInputError with both types of information

This is a MINOR issue, and the fix is low-risk: it only changes the error message content,
not the control flow.

## Risks and edge cases

- The change only affects the error message wording, not when errors are raised.
- Both conflict and failure are already correctly detected at the shell level.
