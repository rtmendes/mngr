# Issue 1: Shell command substitution strips trailing newlines in error accumulation

## What the issue is

In `_apply_work_dir_extra_paths` (host.py, lines 1614 and 1618), the batched symlink
creation script accumulates errors in a shell variable using:

```
errors="${errors}$(printf 'CONFLICT: %s\n' {t})"
```

POSIX command substitution `$(...)` strips all trailing newlines from the output.
`printf 'CONFLICT: %s\n' path` outputs `CONFLICT: path\n`, but `$(...)` strips
that trailing newline, yielding `CONFLICT: path`. When multiple errors accumulate,
they concatenate without any separator:

```
CONFLICT: /path1CONFLICT: /path2
```

The Python-side parsing on line 1624 (`result.stderr.strip().split("\n")`) expects
one entry per line, so it produces a single string. The `removeprefix("CONFLICT: ")`
on line 1625 then yields `/path1CONFLICT: /path2` as a single conflict path.

## Why it is a problem

When a user has multiple paths configured in `work_dir_extra_paths` that conflict
(e.g., multiple non-symlink targets already exist), the error message is garbled
and misleading. Instead of reporting each conflicting path separately, they are
concatenated into a single unintelligible string.

## Planned fix

Replace the error accumulation strategy: instead of building up a shell variable
through command substitution (which strips newlines), write errors directly to
stderr and use a flag variable to track whether any errors occurred.

Specifically:
1. Replace `errors=""` with `had_errors=0`
2. Replace `errors="${errors}$(printf 'CONFLICT: %s\n' {t})"` with
   `printf 'CONFLICT: %s\n' {t} >&2; had_errors=1`
3. Replace `errors="${errors}$(printf 'FAILED: %s\n' {t})"` with
   `{ printf 'FAILED: %s\n' {t} >&2; had_errors=1; }`
4. Replace the final error check with `[ "$had_errors" = 0 ]`

This approach writes errors directly to stderr (no command substitution involved),
preserving newlines. The existing Python-side parsing code on lines 1623-1630
reads `result.stderr`, which will correctly contain the newline-separated error
messages.

## Risks and edge cases

- The `{t}` values are already `shlex.quote`d, so they're safe in printf arguments.
- The curly braces in the `|| { ...; }` for the FAILED case need a space after `{`
  and a semicolon before `}` for POSIX sh compatibility -- this is standard.
- The `had_errors=1` assignment after `printf ... >&2` in the CONFLICT case works
  because both are simple statements separated by `;`.
- Edge case: if the execute_command itself fails for a non-script reason, the exit
  code will still be non-zero and `result.success` will be False, which is handled
  by the existing fallthrough on line 1630.
